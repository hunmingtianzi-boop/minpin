# 监控、备份与恢复运行手册

版本：V1.0  <br>
更新：2026-07-11  <br>
适用范围：API、PostgreSQL、Redis、异步 Worker、Embedding 与大模型调用

## 1. 目标与不可替代的验收证据

MVP 的数据恢复目标为 **RPO ≤ 24 小时、RTO ≤ 4 小时**，PostgreSQL 逻辑备份默认滚动保留 30 天。迁移前备份、每日定时备份、异地对象存储复制、备份失败告警和恢复演练缺一不可。数据库镜像、只检查备份文件存在、在原生产库执行 `pg_restore` 都不算恢复演练。

一次有效演练必须留下机器可读的 `restore-rehearsal-*.json`，至少包含备份 SHA-256、备份年龄、恢复耗时、Alembic 版本、租户/企业/名片/知识/Outbox 行数和最终质量门结果。CI 的临时 PostgreSQL 演练验证脚本和迁移兼容性；生产上线前仍须在与生产同规格、隔离的恢复环境再做一次容量演练。

## 2. 指标入口与数据边界

API 暴露不进入 OpenAPI 的内部端点 `GET /api/v1/metrics`。本地/测试可无鉴权抓取；staging 和 production 启动配置强制要求 `METRICS_BEARER_TOKEN`，抓取端使用 `Authorization: Bearer ...`。凭证只能来自 Secret Manager，不进入镜像、仓库、监控标签或命令历史。

指标只使用方法、路由模板、状态类别、模型、Provider、结果和固定错误码等低基数标签。禁止把 tenant/company/card/visitor/request id、问题文本、手机号、邮箱、IP 或文档内容放入指标。

当前 API 指标包括：

| 指标 | 含义 |
|---|---|
| `cf_http_requests_total` | 按路由模板和状态类别聚合的请求量/错误量 |
| `cf_http_request_duration_seconds` | 包含 SSE 完整响应生命周期的请求延迟直方图 |
| `cf_http_requests_in_flight` | API 当前并发 |
| `cf_db_pool_connections` | 数据库池大小、已借出连接和溢出连接 |
| `cf_ai_requests_total` / `cf_ai_errors_total` | AI 成功、拒答和失败分类 |
| `cf_ai_request_duration_seconds` / `cf_ai_provider_duration_seconds` | RAG 全链路与模型调用延迟 |
| `cf_ai_time_to_first_content_seconds` | 访客从 SSE 开始到首段回答的可感知延迟 |
| `cf_ai_tokens_total` / `cf_ai_estimated_cost_cny_total` | 输入输出 Token 与按配置单价估算的成本 |
| `cf_ai_refusals_total` | 禁答、安全、证据不足等固定拒答码 |
| `cf_rag_retrieval_results` / `cf_rag_citations` | 检索候选和最终引用数量分布 |

Worker 的 `/health/live` 和 `/health/ready` 用于进程/依赖探测；队列积压、租约超时、重试和死信以 Worker 运行手册中的数据库查询及告警为准。任何监控查询都使用只读运维角色，不得复用 Web 应用角色绕过 RLS。

## 3. 最小告警集

阈值是首期默认值，试点压测后按真实基线调整，但不能静默放宽发布门槛。

| 告警 | 初始条件 | 等级 | 首要动作 |
|---|---|---|---|
| API 错误率 | 5 分钟 5xx > 2%，且请求数 ≥ 50 | P1 | 按路由/发布版本定位，必要时停止放量 |
| 核心接口不可用 | 名片、登录、留资连续 3 次探测失败 | P0 | 回滚应用；数据库变更优先向前修复 |
| AI 首内容慢 | 10 分钟 P95 > 5 秒 | P1 | 检查队列、Provider、检索和并发阈值 |
| AI 完整回答慢 | 10 分钟 P95 > 10 秒 | P1 | 降低输出上限或切换经评测的降级模型 |
| AI 失败 | 10 分钟失败率 > 5% | P1 | 检查 Provider 限流/凭证，启用安全降级 |
| 成本异常 | 当日估算成本达到企业预算 80%/100% | P1/P0 | 80% 通知；100% 停止新增模型调用 |
| 数据库池耗尽 | checked_out 接近 size+overflow 持续 5 分钟 | P1 | 查慢 SQL、泄漏连接和异常流量 |
| Outbox 积压 | 最老 pending 超过 5 分钟或持续增长 | P1 | 检查 Worker/Redis；不得直接删除事件 |
| Dead letter | 新增任何死信 | P1 | 按事件类型处置并留人工重放审计 |
| 备份失败/过期 | 定时任务失败或最近成功备份 > 24 小时 | P0 | 立即补备份，冻结破坏性发布 |

PromQL 示例（部署方按采集 job 标签补充筛选）：

```promql
sum(rate(cf_http_requests_total{status_class="5xx"}[5m]))
/
clamp_min(sum(rate(cf_http_requests_total[5m])), 0.001)

histogram_quantile(
  0.95,
  sum by (le) (rate(cf_ai_time_to_first_content_seconds_bucket{source="generated"}[10m]))
)
```

## 4. PostgreSQL 备份

生产调度建议每天 UTC 18:00（北京时间次日 02:00）运行，并在每次数据库迁移前额外运行一次。脚本输出 custom-format dump 和同名 manifest，先写 `.partial`，成功后原子改名，使用 SHA-256 校验，并只清理明确备份目录内超过保留期的匹配文件。

```powershell
$env:MIGRATION_DATABASE_URL = '<从 Secret Manager 注入的 owner/migration URL>'
pwsh -File tools/ops/postgres-backup.ps1 `
  -OutputDirectory D:\secure-backups\cf-ai-card `
  -RetentionDays 30
```

密码通过临时 `PGPASSWORD` 传递，不进入 `pg_dump` 参数。任务完成后，应把 `.dump` 与 `.manifest.json` 复制到启用服务端加密、版本控制/对象锁、跨故障域复制的私有对象存储；上传成功后再记录本次备份成功。访问日志和对象存储生命周期策略也属于审计证据。

## 5. 隔离恢复演练

先由 DBA 创建一个空数据库，名称必须以 `_restore_test` 或 `_restore_rehearsal` 结尾。脚本会校验该后缀，避免误清理生产库；恢复使用单事务、`--clean --if-exists`、不恢复 owner/ACL，并在结束后核验关键表。

```powershell
$env:RESTORE_DATABASE_URL = '<隔离恢复库 URL，例如 .../cf_ai_card_restore_test>'
pwsh -File tools/ops/postgres-restore-rehearsal.ps1 `
  -BackupFile D:\secure-backups\cf-ai-card\cf-ai-card-20260711T180000Z.dump `
  -ExpectedAlembicVersion 20260711_0008 `
  -RpoObjectiveHours 24 `
  -RtoObjectiveSeconds 14400
```

恢复后必须继续执行：

1. 使用只读核验账号抽查两家企业的名片、知识版本、线索密文和审计链，不导出明文 PII。
2. 以应用角色跑 A/B 租户隔离测试；以 Worker 角色跑一次 Outbox 幂等处理。
3. 运行登录、公开名片、AI 固定题、授权留资和隐私请求冒烟。
4. 将 JSON 报告、测试日志、操作者、审批人和环境规格关联到发布单。
5. 销毁隔离恢复库；不得让恢复环境接收真实外部流量或发送真实通知。

## 6. 真实事故恢复顺序

1. 宣布事件、冻结写入和发布，保全日志/审计证据，确认数据破坏时间点。
2. 选择破坏前最近的完整备份；如果生产启用 WAL 归档，在隔离环境做 PITR 并停在破坏前。
3. 校验备份 SHA-256，在新数据库实例恢复，不直接覆盖故障实例。
4. 执行第 5 节的数据、RLS、AI 和业务冒烟；由技术负责人和数据负责人双人确认。
5. 切换应用连接，逐步恢复流量，持续观察错误率、延迟、Outbox 和成本。
6. 记录实际 RPO/RTO、丢失范围、补偿方案和根因；P0 事件完成复盘后才能关闭。

对已批准的数据删除请求：在线库按隐私流程删除/匿名化；不可变备份在 30 天生命周期到期后自然清除。若事故恢复了较旧备份，必须在对外开放前重放删除台账，防止已删除 PII 回流。

## 7. 发布门禁

满足以下条件才能把“监控与灾备”标为通过：

- staging/production 的 metrics 端点鉴权、TLS 和网络访问控制已验证；未授权请求返回 401。
- 仪表盘覆盖 API、数据库、AI/RAG、Worker/Outbox、索引失败和成本；告警有实际接收人。
- 最近成功备份不超过 24 小时，异地副本可读，manifest 校验通过。
- 最近一次隔离恢复 JSON 报告 `passed=true`，实际 RTO ≤ 4 小时。
- 恢复后的 RLS、核心业务、AI 固定题、Worker 幂等和删除台账重放均通过。
- 所有证据关联到发布版本、数据库迁移 head 和评测数据集版本。
