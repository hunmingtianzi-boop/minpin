# 08 架构决策与待确认事项

版本：V1.0  
说明：下列“已定”是开发基线默认值；涉及商务、主体、合规和预算的事项仍需有权限的负责人签署。

## 1. 已定决策（ADR 摘要）

| ADR | 决策 | 状态 | 理由/后果 |
|---|---|---|---|
| ADR-001 | 产品定义为企业级 AI 商务接待平台 | 已定 | 名片是入口；知识、对话、线索和运营才是核心 |
| ADR-002 | V0.1 Demo、V1.0 MVP、V1.5 试点、V2 商业化分期 | 已定 | 消除“第一阶段”混用，避免一次性承诺完整 AI-SCRM |
| ADR-003 | 首发响应式 H5 + Web 管理后台，小程序后置 | 已定 | 扫码可达且交付最快；后端稳定后再复用契约接小程序 |
| ADR-004 | 模块化单体 API + 独立异步 Worker | 已定 | 保持开发速度；按真实瓶颈拆分，不提前微服务化 |
| ADR-005 | React/TypeScript 前端，Python/FastAPI 后端 | 已定 | 冻结原稿中的 React/Vue、FastAPI/NestJS 二选一 |
| ADR-006 | PostgreSQL 18 + pgvector 作为 MVP 数据底座 | 已定 | 事务、全文、向量同库，降低首期运维和一致性成本 |
| ADR-007 | Redis + Celery 处理索引、纪要、通知和评测 | 已定 | 耗时任务与在线请求分离；任务按至少一次投递设计 |
| ADR-008 | AI 使用可控工作流，不使用复杂自治多 Agent | 已定 | 当前场景可由意图、检索、生成、拒答、留资状态机覆盖 |
| ADR-009 | “自学习”=人工审核的知识版本与重索引 | 已定 | 禁止访客原话/AI 草稿自动污染企业知识 |
| ADR-010 | 多租户从首张表实施，应用作用域 + PostgreSQL RLS | 已定 | 多企业泄露是最高风险，不能到试点时补丁式改造 |
| ADR-011 | 回答引用后台保存为 P0，访客端显示为 V1.5 | 已定 | 首期先满足复现和审计，避免公开内部/不可见来源 |
| ADR-012 | MVP 站内通知为事实源，外部消息走 Provider 后置 | 已定 | 微信/企微主体、授权和送达能力未确认 |
| ADR-013 | L1 头像/语气可做，语音/视频数字人后置且不自研 | 已定 | 商业价值在知识接待与线索，不在视频皮肤 |
| ADR-014 | 结构化表单优先，文件解析仅作审核前辅助导入 | 已定 | 资料质量比“支持所有格式”更影响 RAG 质量 |
| ADR-015 | 匿名行为与实名留资分域，企业知识默认不用于模型训练 | 已定 | 数据最小化和供应商风险控制 |
| ADR-016 | OpenAPI 3.1 + SSE；Outbox 传递异步事件 | 已定 | 契约可生成、流式体验、事务与任务可靠交接 |

## 2. 三份源文档冲突裁决

| 冲突 | 源文档表现 | 本基线裁决 |
|---|---|---|
| “第一阶段” | Demo、MVP、Sprint 1-3/4、50 人/50 企业混用 | Demo 仅演示；V1.0 是正式一期；V1.5 承担批量和商会试点 |
| 多企业 | 技术目标要求，但又在后续 Sprint 新增 | 数据模型/RLS Day 1；批量开通和商会 UI 后置 |
| 消息推送 | 原需求多渠道 P0；评审建议站内先行 | 站内 P0；外部渠道需主体/授权，默认 V1.5 |
| 对话导出 | “对话回溯 P0”含导出，另有 P1 数据导出 | MVP 查看/搜索 P0；批量文件导出统一 P1 |
| 回答来源 | 技术目标要求可追溯，来源展示在后续 | 后台保存引用 P0；访客端来源展示 P1 |
| 知识缺口 | Agent 首期任务包含，但试点阶段又列为新增 | V1.0 完成人工审核闭环；V1.5 做聚合运营和趋势 |
| H5/小程序 | 原稿同时出现 | H5 首发；小程序不阻塞 MVP |
| 自学习 | 容易理解成自动训练/吸收 | 只做人工审核和版本化重索引 |
| 数字人 | 市场概念较重 | L1 可选，语音/视频不进入核心路径 |
| 技术栈 | 多个“或”选项 | 本基线冻结 React + FastAPI + Celery + PostgreSQL/pgvector |
| 平台管理员数据权 | 原权限表可看全部 | 默认不可读企业私密正文；只允许受审计 break-glass |

| Q-003 | 管理端首发登录方式？ | 平台预开账号 + 手机验证码或高强度临时密码；微信授权后置 | 阻断 identity 细节；Sprint 0 | 产品/技术 |
| Q-004 | 能否按时提供样板企业结构化资料？ | 3 产品、3 案例、20 FAQ、禁答范围、名片主人信息和视觉素材 | 无资料就无法验收 AI；Sprint 0 第 1 天 | 甲方/产品 |
| Q-005 | 试点口径是 50 人还是 50 家企业？名单是否真实落实？ | 技术容量按 50 家，商务验收暂不承诺 | 阻断 V1.5 排期和容量报告；MVP 立项前 | 业务负责人 |
| Q-006 | MVP 是否强制外部消息推送？若是哪个唯一渠道？ | 不强制，站内后台 P0；外部渠道 V1.5 | 若强制需主体授权、订阅机制、送达/重试，排期增加 | 产品/甲方 |
| Q-007 | 访客数据、企业知识和模型输入的权责/使用边界？ | 企业拥有其知识和业务数据；平台受托处理；默认不训练 | 阻断隐私政策、合同和 Provider 配置；Sprint 0 | 法务/商务 |
| Q-008 | 对话、留资、审计和备份保留期？ | 采用数据文档中的建议初值，支持企业配置但有安全上下限 | 阻断清理任务、隐私政策和存储估算；上线前 | 法务/产品 |
| Q-009 | AI 标识、隐私、同意、删除/投诉流程由谁批准和处理？ | 指定一名合规 Owner 和业务处理 SLA | 阻断公众上线；MVP UAT 前 | 法务/运营 |
| Q-010 | 实际研发团队与每周可用产能？ | 至少 1 FE + 2 BE/AI + 1 QA + 产品/设计支持 | 决定 3-4 周是否成立；Sprint 0 | 项目负责人 |
| Q-011 | MVP 成功标准与严重事故阈值？ | 采用本基线质量/安全门槛，另补有效对话率、留资率、纪要可用率目标 | 决定验收和是否进 V1.5；开发开始前 | 产品/甲方 |
| Q-012 | 平台支持人员是否可以读取企业私密数据？ | 默认不可以；只用限时 break-glass，需企业授权/工单和审计 | 决定权限、客服流程和合同；Sprint 1 前 | 法务/技术 |

## 5. 决策记录模板

每个新 ADR 至少包含：

```text
标题 / 状态 / 日期 / 决策人
背景与约束
候选方案
决策及理由
安全、数据、成本、迁移和运维后果
回滚/替代条件
关联需求、接口、表、测试和负责人
```

仅在会议中口头确认不算关闭。P0 问题需在本文或对应 ADR 中记录决策人、日期和影响范围。
