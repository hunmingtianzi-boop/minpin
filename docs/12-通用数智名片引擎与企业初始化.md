# 通用数智名片引擎与企业初始化

> 文档状态：架构纠偏基线  
> 版本日期：2026-07-10  
> 适用范围：访客 H5、企业内容包、知识问答、后续多租户发布链路

## 1. 纠偏结论

产品的开发顺序必须固定为：

`通用渲染器 -> 企业内容包 -> 运行时 tenant 解析 -> 企业实例发布`

拓浙 AI 生态不是页面架构，也不是产品默认业务模型。根地址与 `/c/template` 展示无真实企业事实的通用模板，拓浙 AI 生态只是第一个用于验证通用能力的企业 seed，tenant key 为 `tuotu`。任何新企业都应以通用模板为结构示例，通过增加内容包、素材和注册记录完成初始化。

本阶段要验证的不是“能否做一个拓浙网站”，而是以下通用闭环：

1. 将企业原始资料整理为结构化、可审核、可追溯的内容包。
2. 用同一套页面组件渲染不同企业的品牌、业务、数据、行动入口和知识问答。
3. 根据路径或查询参数解析 tenant，并只加载静态注册表中的对应内容包。
4. 对缺失字段、未知租户、未审核内容和资料外问题提供安全降级。
5. 明确当前交付是前端静态内容包、运行时校验和本地资料问答，不把规划能力写成已经实现。

## 2. 分层与职责

### 2.1 通用渲染器

通用渲染器负责页面结构和交互，不持有任何企业事实。它包含：

- 响应式页面骨架、导航、首屏、业务模块、数据区、路径区、案例或素材区、FAQ、联系入口和页脚。
- 主题令牌、布局变体、图标、动效、无障碍与减少动态效果支持。
- 内容缺省态、未知租户页、素材加载失败态和可选模块隐藏规则。
- AI 助手入口、引用展示、拒答、转人工与会话 UI。

禁止在这一层出现 `tuotu`、某家企业名称、某个企业资产路径或某个企业专用问答规则。

### 2.2 企业内容包

企业内容包是可运行配置，不是页面源码。当前 `EnterpriseCardConfig` 至少包含：

- `id`、`version`：tenant key 与内容版本。
- `seo`：页面标题和描述。
- `brand`：品牌名称、定位文案、Logo、官网行动和无障碍标签。
- `theme`：深浅主题、行动颜色、首屏遮罩和圆角令牌。
- `hero`：首屏文案、图片、行动和指标。
- `sections`：页面 block 数组。
- `assistant`：本地知识、来源、快捷问题、交互标签和拒答内容。
- `footer`：品牌说明、免责声明和返回顶部行动。

事实内容与展示配置应分开。企业可以调整视觉表达，但不能通过视觉配置绕过事实审核和禁答规则。

当前内容包用 `sections` block 数组控制页面内容和顺序。渲染器支持 `feature-grid`、`media-showcase`、`process`、`evidence`、`engagement`、`faq` 和 `closing` 7 种 block。每种 block 的字段与数量限制见 [企业内容包模板使用说明](13-企业内容包模板使用说明.md)。

### 2.3 tenant 解析器

tenant 解析器只负责识别当前企业并返回已发布内容包，不负责页面展示。MVP 的解析优先级为：

1. 路径参数，例如 `/c/template` 或 `/c/tuotu`。
2. 查询参数，例如 `/?tenant=tuotu`。
3. 没有显式 tenant 时加载 `template`。

解析器只接受注册表中的 tenant key。未知 tenant 不得回退到 `template`、`tuotu` 或其他企业，应显示安全的“名片不存在或尚未发布”状态。

### 2.4 通用模板

`template` 是真正可运行的无品牌通用内容包。根地址和 `/c/template` 都加载该配置。它完整展示 7 类 block、企业资料助手、主题令牌、内容审核提示和素材目录结构。

模板中的企业身份、业务描述、示意数据、`example.com`、图片和问答只用于说明结构。它们不是任何企业事实，不得直接上线。接入新企业时以 `src/tenants/template/tenant.ts` 为完整示例，但只复制并修改内容包、素材目录和注册记录，不修改 `App.tsx` 或 `styles.css`。

### 2.5 企业 seed

seed 是用于初始化、演示和自动化测试的首批企业数据。`tuotu` seed 的职责是：

- 验证内容包能承载真实企业材料和不确定信息。
- 验证通用模块可处理品牌、三项业务、数据指标、活动图片和受控 FAQ。
- 验证问答引用、资料边界和拒答规则。
- 作为新增企业内容包的结构示例，不作为视觉或业务模板的唯一范式。

生产环境不得把企业 seed 当作平台默认企业。当前入口未携带 tenant 时展示无真实企业事实的 `template`，显式请求未知 tenant 时进入无品牌错误状态。

## 3. 目标目录

前端当前目录如下：

```text
apps/card-web/
  src/
    domain/
      card.ts                # 内容包、主题、模块、行动和知识类型
    components/              # 企业无关的主题与 AI 助手 UI
    lib/
      knowledge.ts           # 通用问答匹配、引用和拒答逻辑
      tenantRuntime.ts       # SEO、主题与租户素材运行时注入
      validateTenantConfig.ts # 企业内容包运行时结构与安全校验
    tenants/
      index.ts               # MVP 注册表、tenant 解析和内容加载
      defineTenant.ts        # 内容包定义与类型约束
      template/
        tenant.ts            # 可复制的完整通用内容包示例
      tuotu/
        tenant.ts            # seed 元数据、内容、主题、知识和策略
    App.tsx                  # 通用页面编排和模块渲染
    main.tsx
    styles.css               # 平台基础令牌与主题映射
  public/
    tenants/
      template/
        assets/              # 模板示意素材，不可直接用于企业发布
      tuotu/
        assets/              # 该租户自有且已授权的素材
```

以下后端目录只是后续规划，不属于当前模板交付：

```text
services/api/modules/
  tenants/                   # 租户、域名、状态和发布版本
  content/                   # 内容草稿、审核、版本和发布
  knowledge/                 # 文档、切片、索引和引用
  conversations/             # 会话、消息、转人工和同意记录
  leads/                     # 留资、分配、通知和状态

packages/contracts/
  openapi.yaml               # 平台 API 契约
  tenant-content.schema.json # 企业内容包版本化 Schema
```

## 4. 数据流

### 4.1 企业初始化流

```text
原始资料
  -> 文档与素材抽取
  -> 事实、宣传语、待确认项和禁答项分类
  -> 编写 TypeScript 内容包
  -> 企业审核人与平台审核人复核内容
  -> 在静态注册表中注册 tenant
  -> 运行测试与构建
  -> 通用渲染器加载并展示
```

原始 DOCX、PDF、网页或图片只作为来源，不直接成为线上事实。抽取结果必须保留来源位置和时间，所有面向用户的事实必须经过审核后写入企业内容包。

### 4.2 页面请求流

```text
访客请求
  -> 解析路径或查询参数
  -> 从静态注册表读取内容包
  -> 运行内容包结构与安全校验
  -> 注入 SEO、主题与素材
  -> 通用渲染器生成页面
```

页面在挂载通用渲染器前完成 tenant 解析与校验，避免先展示其他企业再切换。

当前根地址直接解析为 `template`，`/c/template` 是它的显式地址，`/c/tuotu` 加载首个企业 seed。未知 tenant 进入无品牌错误状态。

### 4.3 AI 问答流

```text
用户问题
  -> 使用当前 tenant 的 knowledgeBase
  -> 本地关键词匹配
  -> 返回已配置答案和来源
  -> 无匹配时返回该 tenant 的 fallback
```

当前版本不调用外部模型，也没有向量检索。问答只读取当前 tenant 的本地知识配置，不得使用其他企业内容补全答案。

## 5. 新企业接入步骤

### 第一步：建立企业档案

分配稳定的 tenant key。key 仅使用小写字母、数字和短横线，发布后不随品牌展示名变化。收集主体关系、Logo、官网、业务、案例、指标、联系入口、隐私政策和素材授权。

### 第二步：建立来源台账

使用 [样板企业资料采集模板](10-样板企业资料采集模板.md) 收集资料。对每条事实记录来源、日期、审核状态和允许使用范围。营销表达与可验证事实分别管理。

### 第三步：生成内容包草稿

复制 `src/tenants/template/tenant.ts` 作为新企业内容包，再把资料映射到现有类型。不存在的 block 可以从 `sections` 中移除，不能用虚构文字填满版面。

### 第四步：配置品牌与素材

在现有主题令牌范围内设置颜色、圆角和深浅主题。素材存入 `public/tenants/{tenant}/assets/`，并记录版权、肖像和品牌授权状态。

### 第五步：配置知识与边界

建立标准 FAQ、同义问法、来源和有效期。单独维护禁答、条件回答、必须转人工和实时系统查询四类规则。价格、合同、资质、医疗、金融、法律等高风险内容必须有专项审核。

### 第六步：预览与双重审核

在预览版本中检查桌面端、移动端、分享卡片、图片替代文本、链接、电话和表单。由企业审核人确认事实，由平台审核人确认合规、隐私、可访问性和跨租户隔离。

### 第七步：发布和绑定入口

在 `src/tenants/index.ts` 增加 import、注册表记录和 export，通过 `/c/{tenant}` 或 `?tenant={tenant}` 预览。当前静态版本上线前必须完成企业审核并运行测试与构建。

### 第八步：持续更新

新资料先完成事实和授权复核，再修改企业内容包并更新 `version`。每次修改后重新运行测试与构建，不能把未确认资料直接写入正式内容包。

## 6. 内容审核与发布门槛

当前静态版本没有内容审核后台和字段状态机，企业与平台审核通过人工清单完成。发布门槛包括：

- 企业展示名、品牌关系和运营主体不产生误导。
- 所有数字指标都有口径、截止日期和证据。
- 产品能力区分已上线、试运行、规划和历史状态。
- 合作方名称、Logo、案例和评价已有公开授权。
- 图片版权、人物肖像、隐私和联系方式使用范围明确。
- FAQ 有来源，禁答和转人工规则已覆盖高风险问题。
- 所有外链、二维码和联系入口真实有效。
- 内容包通过运行时结构校验和现有自动化测试。
- 页面不存在其他租户名称、素材或知识引用。

当前至少要在企业资料台账中保存来源、审核人、审核时间和版本说明。AI 可以辅助抽取和生成草稿，但不能替代企业事实确认或平台发布审批。

## 7. MVP 实现约束

- MVP 可使用前端静态注册表和 TypeScript 内容包，但页面组件必须只依赖通用类型。
- `template` 是根地址和 `/c/template` 展示的通用模板，不含真实企业事实。
- `tuotu` 是首个企业 seed，访问方式为 `/?tenant=tuotu` 或 `/c/tuotu`。
- 页面模块由带 `type` 判别字段的 block 数组声明，模块顺序和显隐由受控配置决定。
- 每次页面加载都根据当前 tenant 注入主题、页面标题、描述、素材和问答配置。
- 本地知识匹配只用于交互验证，仍需带 tenant 参数和来源信息。
- 禁止通过复制 `App.tsx`、复制整套 CSS 或按企业建立独立构建工程来接入新企业。
- 新企业以 `src/tenants/template/tenant.ts` 为完整示例，只修改内容包、租户素材与 `src/tenants/index.ts` 注册记录。
- 禁止把企业事实写进通用组件的默认值。默认值只能是无品牌的安全占位或错误状态。
- 模板占位内容不得作为企业正式内容发布。

## 8. 生产演进规划

本节是后续路线图，不属于当前通用模板已经实现的功能。当前已交付范围仅为阶段 A。

### 阶段 A：静态多租户

- TypeScript 内容包与静态素材。
- 前端 tenant 注册表。
- 通用渲染器、主题和本地 FAQ。
- 根地址与 `/c/template` 通用模板。
- `/c/{tenant}` 与 `?tenant={tenant}` 入口。
- 运行时内容包结构校验，以及 tenant 注册表和校验器测试。

### 阶段 B：内容后台与版本发布

- PostgreSQL 保存企业、内容草稿、审核记录和发布版本。
- 对象存储保存租户素材，使用独立前缀和签名访问。
- 后台支持预览、字段级审核、定时发布和回滚。
- 域名映射、分享配置和多环境发布。

### 阶段 C：生产 RAG 与线索闭环

- 文档解析、切片、向量索引和版本化知识库。
- SSE 对话、引用、低置信度拒答、转人工和评测集。
- 访客同意、留资、销售通知、分配和状态跟进。
- 租户级用量、成本、质量、敏感事件和审计报表。

### 阶段 D：规模化企业生成

- 资料上传后自动形成内容草稿、缺失项和风险项清单。
- 行业内容模型、模块组合、审核策略和设计主题预设。
- 自定义域名、团队权限、计费、套餐和发布 SLA。
- 灰度发布、缓存隔离、区域部署、灾备和企业级安全能力。

## 9. 验收标准

架构纠偏完成需同时满足：

1. 根地址与 `/c/template` 展示无真实企业品牌的通用模板。
2. 通用模板完整展示 7 类 block，并明确所有占位内容不可直接上线。
3. 通用组件和渲染器不导入 `tuotu` 内容模块。
4. `tuotu` 可以通过 `/?tenant=tuotu` 和 `/c/tuotu` 访问。
5. 未知 tenant 显示无品牌安全状态，不展示模板或拓浙内容。
6. 新增企业只增加内容包、素材和注册记录，不修改 `App.tsx` 或 `styles.css`。
7. 切换 tenant 后标题、主题、图片、FAQ、引用和会话上下文全部随之切换。
8. 测试和构建可以发现内容包错误和跨租户内容泄漏。
9. 拓浙内容包的事实、来源、禁答和授权待确认项仍以 [拓浙 AI 生态 seed 内容包](11-拓浙AI生态样板企业资料包.md) 为准。

## 10. 当前模板与 seed

- `template`：无真实企业品牌的通用模板，入口为根地址和 `/c/template`。其占位内容不得直接上线。
- `tuotu`：首个企业 seed，入口为 `/c/tuotu`。资料来自用户提供的三份规划文档和拓途浙享公开网站。

原始 DOCX 保持未修改。`tuotu` seed 内容仍处于内部演示状态，不能替代项目方的正式发布审核。
