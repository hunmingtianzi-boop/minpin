# 15 企业管理后台与知识发布

版本：V0.1，2026-07-11
状态：本地可运行、可联调、可验收

## 1. 目标与边界

企业管理后台是通用平台能力，不绑定 `tuotu`。任何已创建租户、企业、名片和员工凭据的企业都复用同一套后台与 API；拓浙资料只作为首个初始化数据包。

V0.1 完成四条主流程：

1. 员工账号登录、刷新、退出和当前身份查询。
2. 编辑企业资料并通过乐观锁避免多人静默覆盖。
3. 编辑名片名称、助手文案、推荐问题和协议版本。
4. 新建或修改知识草稿，生成 1024 维向量，成功后原子发布并立即供公开 H5 与 AI 问答使用。

暂不包含文件批量解析、组织成员邀请、完整 RBAC 配置器、知识删除/归档、回滚 UI、线上监控大盘和 CRM。

## 2. 本地入口

```powershell
pnpm local:start
pnpm local:status
```

| 服务 | 地址 |
|---|---|
| 企业管理后台 | `http://127.0.0.1:4174/` |
| 访客 H5 | `http://127.0.0.1:4173/c/tuotu` |
| FastAPI / OpenAPI | `http://127.0.0.1:8000/docs` |
| Embedding API | `http://127.0.0.1:8010/v1` |

首次导入管理员需要在被 Git 忽略的 `.env.local` 配置：

```dotenv
ADMIN_BOOTSTRAP_TENANT_SLUG=企业租户slug
ADMIN_BOOTSTRAP_ACCOUNT=管理员账号
ADMIN_BOOTSTRAP_PASSWORD=仅本地或密钥系统提供的强密码
```

初始化凭据只在不存在时创建，重复启动不会重置密码。生产环境不得使用仓库默认值或把密码写进前端变量。

## 3. 身份与安全

- 密码使用带随机盐的 `scrypt` 保存，数据库不保存明文。
- 登录失败使用统一错误，并记录失败次数和定时锁定，避免账号枚举。
- Access Token、Refresh Token 和访客 Token 使用不同 `typ/aud`，不能互换。
- Refresh Token 仅保存 SHA-256 摘要，每次刷新旋转；重放会撤销整个会话。
- 退出后服务端立即撤销会话，旧 Access Token 不能继续操作后台。
- 员工身份固定绑定 `user + membership + tenant + company`，业务请求不能从浏览器参数切换数据作用域。
- 后台读写同时执行应用层 scope 条件和 PostgreSQL 强制 RLS。
- `company_admin` 与 `platform_admin` 具有当前纵向切片权限；其他角色必须持有对应显式权限。

## 4. API 契约

统一前缀：`/api/v1`。

| 方法与路径 | 作用 | 关键约束 |
|---|---|---|
| `POST /auth/login` | 员工登录 | `{account, credential, method: "password"}` |
| `POST /auth/refresh` | 刷新并旋转会话 | 旧 Refresh Token 立即失效 |
| `POST /auth/logout` | 撤销会话 | Bearer Staff Access Token |
| `GET /auth/me` | 当前员工、成员关系和企业 | 校验会话仍有效 |
| `GET/PUT /admin/company/profile` | 企业资料 | PUT 必须带 `If-Match` |
| `GET/PUT /admin/card` | 名片与 AI 助手展示配置 | PUT 必须带 `If-Match` |
| `GET/POST /admin/knowledge/documents` | 知识列表/建立空文档 | POST 后继续 PUT 草稿 |
| `GET /admin/knowledge/documents/{id}` | 获取可编辑原文和元数据 | 不从列表摘要猜测正文 |
| `PUT /admin/knowledge/documents/{id}` | 建立不可变草稿版本 | 原发布版本继续在线 |
| `POST /admin/knowledge/documents/{id}/publish` | 索引并发布最新草稿 | 向量全部成功后才切换 |

OpenAPI 事实源是 `packages/contracts/openapi.json`。后端变更后运行：

```powershell
pnpm contracts:sync
pnpm contracts:validate
```

## 5. 知识发布事务

```text
保存草稿
  → 新建不可变 KnowledgeVersion
  → 切片并保持 is_active=false
  → 事务提交
  → 事务外批量生成 passage: 向量
  → 校验批次数、1024 维和有限数值
  → 新事务锁定文档/草稿/索引任务
  → 写入全部向量
  → 旧块失活、新块生效
  → 原子切换 current_version_id
  → 审计日志落库
```

Embedding 超时、维度不符、任务取消或并发版本冲突时，索引任务记录失败，旧发布版本保持在线，不允许半成品进入检索。

每次企业资料、名片、草稿和发布操作都记录带前序哈希的审计条目。该哈希链用于发现日志被篡改，不替代数据库备份或外部不可变审计存储。

## 6. 初始化资料与后台数据的优先级

企业资料包只负责首次初始化。服务重启时：

- 不覆盖后台修改过的企业名称、简介、行业、区域和网站。
- 不覆盖后台修改过的名片与助手配置。
- 不重置员工密码、成员状态或权限。
- 不把管理员已发布的新知识切回种子版本。
- 不重新激活旧种子向量，避免公开问答同时检索到两个版本。

后续如需再次导入资料包，应通过显式导入与审核流程，而不是借助应用重启隐式覆盖。

## 7. 前端状态与可用性要求

后台使用 Fluent UI 企业组件体系，保持浅色、高密度、键盘可用和清晰的操作反馈。每个数据页面必须覆盖：

- 首次加载与按钮提交中的忙碌状态。
- 空列表和首次创建引导。
- 401 自动刷新一次，刷新失败回登录页。
- 403 无权限、404 资源不存在、409 版本冲突和 5xx 服务错误。
- 草稿、索引中、发布成功、索引失败等状态标签。
- 发布前明确确认，发布中禁止重复提交。

Access Token 只保存在内存，Refresh Token 使用当前版本的 `sessionStorage` 以避免跨浏览器重启长期驻留。正式上线前应改为 Secure、HttpOnly、SameSite Cookie 并加入 CSRF 防护。
