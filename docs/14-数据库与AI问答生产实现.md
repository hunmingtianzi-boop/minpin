# 14 数据库与 AI 问答生产实现

版本：V0.2（2026-07-10）  
状态：本地完整链路已验收；生产发布仍需试点评测、监控与灾备验收

## 1. 本阶段结论

数据库和 AI 问答是产品主干，不再沿用浏览器内关键词匹配作为正式能力。生产路径统一为：

```text
公开名片
  → 访客短期会话与同意记录
  → 企业/名片作用域校验
  → PostgreSQL RLS
  → 当前已发布知识版本
  → 混合检索与证据门控
  → DeepSeek 结构化生成
  → 输出安全检查
  → 消息、AI run、引用、用量原子落库
  → SSE 返回访客
```

当前浏览器静态问答仅作为“未配置 API 的模板演示降级”，不得用于正式企业知识、内部资料或联系人数据。

## 2. 技术基线

| 层 | 实现 |
|---|---|
| API | Python 3.12/3.13、FastAPI、Pydantic 2 |
| 数据访问 | SQLAlchemy 2 async、Alembic |
| 数据库 | PostgreSQL 16+、pgvector 0.8.x、`pg_trgm`；本机已验证 16.14 / 0.8.3 |
| 缓存/限流 | Redis |
| 生成模型 | DeepSeek OpenAI-compatible Chat Completions |
| 默认模型 | `deepseek-v4-flash`，`thinking.type=disabled` |
| 向量模型 | 本地 `intfloat/multilingual-e5-large`、1024 维、FastEmbed CPU、OpenAI-compatible 接口 |
| 检索 | PostgreSQL 全文/三元组 + pgvector + RRF；向量服务故障时受控降级为词法检索 |
| 输出 | 严格 JSON 内部结果，校验后转 SSE |
| 可观测 | request/trace id、AI run、模型/Prompt/知识版本、token、延迟、错误分类 |

DeepSeek 官方已将 V4 作为当前 API 模型，并公告 `deepseek-chat`、`deepseek-reasoner` 将于 2026-07-24 15:59 UTC 下线。代码必须显式选择 V4 模型并显式设置思考开关，不能依赖即将失效的别名。参见 [DeepSeek API 首次调用](https://api-docs.deepseek.com/) 与 [DeepSeek V4 发布说明](https://api-docs.deepseek.com/news/news260424/)。

## 3. 密钥规则

1. 模型密钥只允许通过服务端 `LLM_API_KEY` 注入。
2. 不得写入 TypeScript、Python、JSON 内容包、数据库业务字段、浏览器环境变量或日志。
3. `VITE_*` 变量会进入浏览器构建产物，严禁保存任何模型密钥。
4. 配置对象使用秘密类型；日志处理器对授权头、Token、Prompt 和密钥字段统一脱敏。
5. 聊天中、工单中或截图中出现过的密钥视为已暴露，联调后立即撤销并创建新密钥。

本地使用被 `.gitignore` 排除的 `.env.local`：

```dotenv
LLM_PROVIDER=deepseek
LLM_BASE_URL=https://api.deepseek.com
LLM_API_KEY=替换为轮换后的新密钥
LLM_MODEL=deepseek-v4-flash
LLM_THINKING=disabled
```

## 4. 数据安全边界

数据库采用四层隔离：

1. 入口从公开 `card_slug` 或签名访客令牌推导企业，客户端不能提交 `company_id` 改变作用域。
2. Use case/repository 必须显式持有 `tenant_id + company_id`。
3. 每个事务通过 `set_config(..., true)` 设置 PostgreSQL 本地作用域；核心表启用并强制 RLS。
4. 检索 SQL 自身同时校验 tenant、company、document current version、review status 和 visibility；禁止先全库召回再在 Python 过滤。

迁移账号与 API 运行账号必须分离。API 账号不得是表所有者、超级用户或拥有 `BYPASSRLS`；本地 Compose 使用受限的 `cf_ai_card_app`，生产应由云数据库身份/密钥系统创建等价角色。

公开 slug 只是路由标识，不是认证凭据。读取公开名片仍要求 slug 精确相等；任何对话、同意和留资写入还必须持有对应名片签发的访客令牌。

## 5. 知识生命周期

```text
企业内容包/审核文件
  → Pydantic Schema 校验
  → KnowledgeDocument
  → 不可变 KnowledgeVersion
  → 结构化切片
  → 可见范围检查
  → 可选 embedding
  → 候选索引抽样检查
  → 原子切换 current_version_id
```

- 没有向量服务时可以使用词法检索完成开发联调，但不能据此宣称达到最终检索门槛。
- 更换 embedding 模型或维度必须构建新索引版本，不能原地混用。
- `app.cli.index_embeddings` 使用“新建影子版本 → 完成全部向量 → 继承审核 → 原子切换”的写时复制流程；失败时旧索引继续在线，禁止半成品进入检索。
- 删除、撤权或归档必须让旧版本立即退出在线检索；后台物理清理按保留策略执行。
- 当前通用模板与拓浙初始资料分别位于 `packages/tenant-content/template.knowledge.json` 和 `packages/tenant-content/tuotu.knowledge.json`。

## 6. 在线问答原则

1. 问候、隐私说明、留资和转人工使用确定性模板，不浪费模型调用。
2. Prompt injection、跨企业探测、个人隐私、高风险建议和未经授权的具体报价先由策略层阻断。
3. 企业事实只有在证据门控通过后才交给模型。
4. 模型只能引用传入的 evidence id；未知或越界引用使整次回答拒绝发布。
5. 内部 JSON 必须经 Pydantic 校验。空 JSON、截断、content filter 和资源不足均映射为安全失败。
6. 对访客只返回可见来源标签；完整片段仅供有权限的后台审计。
7. 每次回答都保存输入/输出哈希、Prompt 版本、模型、检索版本、引用、token、耗时和错误类别，不保存模型思考链。

## 7. 本地运行

当前 Windows 开发机已经配置 PostgreSQL/pgvector、Redis、Embedding、API、访客 H5 和企业管理后台六个本地服务。日常使用：

```powershell
pnpm local:start
pnpm local:status
pnpm local:stop
```

`local:start` 会自动执行迁移、首次内容初始化和安全向量重建。初始化资料不会覆盖管理员已修改或发布的数据。首次启动如本地没有模型，会下载约 2 GB 的多语种 E5 模型；后续直接复用 `%LOCALAPPDATA%\cf-ai-card-runtime\models` 缓存。所有服务仅监听 `127.0.0.1`。

其他开发机的手工/容器方式前置条件：Docker/兼容容器运行时、Node.js 24、Python 3.12 或 3.13。

```powershell
Copy-Item .env.example .env.local
# 只在 .env.local 中填写轮换后的服务端密钥
docker compose --env-file .env.local -f infra/compose.yaml up -d postgres redis minio

python -m venv services/api/.venv
services/api/.venv/Scripts/python -m pip install -e "services/api[dev]"
services/api/.venv/Scripts/python -m alembic -c services/api/alembic.ini upgrade head
services/api/.venv/Scripts/python -m uvicorn app.main:app --app-dir services/api --host 127.0.0.1 --port 8000

corepack pnpm --filter @cf/card-web dev
```

本机未安装 Docker 时仍可运行不依赖数据库的单元测试，但不能完成 RLS、pgvector、迁移和真实检索验收。

## 8. 必过测试

```text
单元：Token、SSE、策略、证据门控、Provider 错误分类、JSON 校验
契约：OpenAPI 路径、状态码、SSE 事件
数据库：迁移正向执行、约束、RLS A/B 企业越权、公开 slug 精确作用域
检索：当前版本、visibility、删除/归档、向量维度、词法降级
模型：超时、401/402/429/5xx、空 JSON、截断、content_filter
端到端：visit → consent → conversation → answer → citation → replay
```

发布硬门槛沿用基线：Retrieval Hit@5 ≥ 85%、有资料回答通过率 ≥ 80%、无资料拒答率 ≥ 90%、引用支持率 100%、跨企业泄露 0、严重 Prompt injection 失败 0、完整回答 P95 ≤ 10 秒。

版本化冒烟集位于 `packages/evals/`。数据库、内容包和轮换后的模型密钥准备完成后运行：

```powershell
services/api/.venv/Scripts/python -m app.cli.evaluate_rag packages/evals/tuotu.v1.json --output artifacts/evals/tuotu-v1.json
services/api/.venv/Scripts/python -m app.cli.evaluate_rag packages/evals/tuotu.v2.json --output artifacts/evals/tuotu-v2.json
```

命令未通过发布门槛时返回非零退出码，可直接作为 CI/发布流水线 gate。自动指标不能替代人工事实、完整性和表达质量复核。

2026-07-10 本地真实链路已运行 `tuotu-v1` 的 13 个用例并通过 gate：Retrieval Hit@5、有效回答交付、正确拒答和预期来源引用均为 100%，严重安全失败为 0，P95 为 2.653 秒。报告保存在 `artifacts/evals/tuotu-v1.json`；扩大企业范围前仍需为每家企业建立独立 golden set。

2026-07-11 在“拓浙 AI 集团”四业务板块与 AI 场景服务知识更新后运行 `tuotu-v2` 的 15 个用例并通过 gate：Retrieval Hit@5、有效回答交付、正确拒答和预期来源引用均为 100%，严重安全失败为 0，P95 为 1.904 秒。报告保存在 `artifacts/evals/tuotu-v2.json`。

## 9. “顶级”验收口径

“接入一个强模型”不等于顶级。只有以下闭环全部存在并有试点数据证明，才能进入成熟产品区间：

- 内容增量同步、删除与权限变更 SLA；
- 数据库与检索的服务端强制隔离；
- hybrid、rerank 和阈值由评测决定；
- 精确引用和无证据拒答；
- Prompt、模型、语料与索引可版本化、可回滚；
- 固定 golden set、CI 回归、线上抽样和人工复核；
- 全链路 trace、成本、延迟、错误与知识缺口看板；
- 故障降级、备份恢复、密钥轮换和应急审计。

当前本地链路已经完成真实 PostgreSQL/RLS、混合检索、本地多语种向量、DeepSeek 生成、SSE 和引用落库验收。它已经从“静态演示”进入可开发基线，但 rerank、试点 golden set 校准、线上监控、备份恢复演练和密钥轮换仍是生产发布门槛，不能以一次成功问答替代正式验收。
