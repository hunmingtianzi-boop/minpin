# 07 开发环境与工程规范

版本：V1.0

## 1. 运行时与依赖策略

| 工具 | 基线 | 规则 |
|---|---|---|
| Node.js | 24 LTS | `.nvmrc`/Volta 固定大版本；CI 与容器一致 |
| pnpm | 11.x | 根 lockfile，禁止混用 npm/yarn |
| Python | 3.13 生产基线；CI 兼容 3.12 | `pyproject.toml` 声明范围，生产镜像固定 patch |
| PostgreSQL | 18.x | 固定最新安全小版本 |
| pgvector | 0.8.x | 镜像与 PostgreSQL 大版本匹配 |
| Redis | 受支持稳定版 | 只作缓存/队列，不作业务事实源 |

应用依赖必须进入 lockfile；容器镜像使用 digest 或明确版本，禁止生产使用 `latest`。依赖升级由自动化 PR 或专门任务完成，必须跑契约、迁移、RAG 和安全回归。

## 2. 仓库结构

```text
apps/
  card-web/
    src/{app,features,entities,shared}
  admin-web/
    src/{app,features,entities,shared}
services/
  api/
    src/cf_ai_card/
      modules/<module>/{domain,application,infrastructure,api}
      shared/{auth,db,events,observability,providers}
    tests/{unit,integration,contract,security}
  worker/
    src/cf_ai_worker/{tasks,consumers}
packages/
  contracts/{openapi.json,events/}
  ui/
db/
  migrations/
infra/
  compose.yaml
  nginx/
docs/
tools/
```

前端按业务 feature 组织，不按 `components/hooks/utils` 大杂烩组织。后端模块公开 service 是边界，禁止跨模块直接 import 另一个模块的 ORM model 并改表。

## 3. 本地开发流程

目标命令（仓库脚手架落地后由根脚本封装）：

```powershell
Copy-Item .env.example .env
docker compose -f infra/compose.yaml up -d postgres redis minio

pnpm install --frozen-lockfile
pnpm dev

python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install -e ".\services\api[dev]"
alembic upgrade head
uvicorn cf_ai_card.main:app --reload
celery -A cf_ai_worker.app worker --loglevel=INFO
```

本机没有 Docker 时，可连接团队 dev 环境，但不得使用生产数据库/密钥进行日常开发。

开发环境必须提供 seed 命令，创建两个企业 A/B、各一张名片和不相同知识，用于每次租户隔离冒烟。

## 4. 环境与配置

环境固定为 `local`、`test`、`staging`、`production`。配置经 Pydantic Settings/前端构建变量读取，并在启动时校验；缺关键配置立即失败。

建议变量分组：

```text
APP_ENV, APP_BASE_URL, API_BASE_URL
DATABASE_URL, REDIS_URL
OBJECT_STORAGE_ENDPOINT/BUCKET/KEY_REFERENCE
JWT_SIGNING_KEY_REFERENCE, FIELD_ENCRYPTION_KEY_REFERENCE
LLM_PROVIDER/BASE_URL/MODEL/API_KEY_REFERENCE
EMBEDDING_PROVIDER/MODEL/DIMENSION/API_KEY_REFERENCE
OTEL_EXPORTER_OTLP_ENDPOINT, SENTRY_DSN
RATE_LIMIT_*, MODEL_BUDGET_*, DATA_RETENTION_*
```

- 仓库只提交 `.env.example` 和假值。
- staging/production 密钥由 KMS/Secret Manager 注入，不通过聊天、邮件或 CI 日志传递。
- 前端构建变量一律视为公开信息，禁止放密钥。
- 每个环境使用独立数据库、对象存储前缀、Redis namespace 和模型密钥。

## 5. Git 与变更流程

- `main` 始终可发布；使用短分支和 Pull Request。
- Commit 建议采用 Conventional Commits：`feat(cards): ...`、`fix(knowledge): ...`。
- 一个 PR 聚焦一个可验证问题；数据库、契约、实现和测试尽量在同一变更中闭环。
- 需要架构/安全/数据迁移判断的变更先写 ADR。
- 禁止 force-push 共享主分支、跳过 CI 合并或直接修改生产数据库。
- 生成代码单独标识；不得在生成文件里手改业务逻辑。

推荐评审人：业务模块代码至少 1 名同域研发；权限/RLS/加密/迁移/模型安全至少 1 名技术负责人或安全 Owner。

## 6. API 与兼容性

- 先改 `packages/contracts/openapi.json`，再生成类型/客户端并实现。
- 向后兼容添加字段；删除/重命名/改变语义属于破坏性变更，需版本或迁移窗口。
- 对枚举增加值也可能破坏前端，必须提供 unknown fallback。
- 请求/响应、事件和 AI 结构化输出都用 Schema 校验。
- 前端不得用 `any` 绕过契约；后端不得返回 ORM 对象自动序列化。

## 7. 数据库迁移

- Alembic migration 是唯一 DDL 来源；应用启动不自动建表。
- migration 文件必须可在空库和上一生产版本上执行。
- 线上采用 expand/contract：先加兼容字段/双读写，再回填，再切换，最后删除旧结构。
- 大表索引、回填和向量重建走可观察后台任务，避免长事务锁表。
- RLS policy、角色和安全函数必须随 migration 管理并有测试。
- 发布前备份；回滚优先使用向前修复，不能假设 DDL 可无损逆转。

## 8. Python 代码约束

- 类型检查覆盖业务 service、Provider 接口和 API Schema。
- 使用 SQLAlchemy 2.0 显式 `select` 和事务边界；Repository 不自行 commit。
- HTTP 路由只做解析、鉴权、调用和映射；业务状态机放 application/domain。
- 不在 async 请求线程执行 CPU 密集解析或批量 embedding；投递 Worker。
- 外部调用必须设置连接/读取超时、可观测重试和熔断。
- 时间从可注入 clock 获取，ID 从统一生成器获取，便于测试。
- 禁止捕获裸 `Exception` 后静默继续；错误映射为稳定内部类别。

建议工具：Ruff（格式/静态检查）、mypy/pyright（类型）、pytest（测试）、Bandit/依赖扫描（安全）。

## 9. 前端代码约束

- TypeScript strict；服务端数据先经过生成类型和运行时边界校验。
- 公开 H5 与管理后台独立入口；公共包只放无业务所有权的 UI/契约。
- 权限控制不能只隐藏按钮；前端仅改善体验，后端是最终授权点。
- AI 对话必须处理 SSE 断开、重连、重复事件、超时、安全拒答和消息最终状态。
- 所有页面具备 loading、empty、error、offline/重试和无权限状态。
- 移动端优先；至少覆盖微信内置浏览器的约定版本、主流 Android/iOS 和桌面后台。
- 可访问性：键盘可用、表单 label、焦点、对比度、动态内容通知。

建议工具：ESLint、Prettier、Vitest、Testing Library、Playwright；性能用 Lighthouse/Web Vitals 在固定环境回归。

## 10. 测试金字塔

| 层 | 范围 | 合并门槛 |
|---|---|---|
| 单元 | 状态机、权限函数、切片、策略、Schema | 快速、确定、无外部网络 |
| 集成 | PostgreSQL/RLS、Redis 队列、对象存储、Provider stub | CI 使用真实依赖容器 |
| 契约 | OpenAPI、事件、SSE、Provider adapter | 前后端与消费者一致 |
| E2E | 企业建档→发布名片→访客问答→留资→主人跟进 | 主路径和关键异常 |
| AI 评测 | 检索、忠实、拒答、注入、纪要 | 模型/Prompt/知识变化必跑 |
| 安全 | IDOR、跨租户、RLS、输入/上传、Token、速率 | 严重失败阻断发布 |

测试默认不调用付费生产模型；使用 deterministic stub。模型真实回归在受控 CI/staging 作业运行，记录费用和版本。

## 11. CI/CD 门槛

PR 阶段：

1. 格式、lint、类型检查。
2. 单元、集成、契约测试。
3. migration 空库/升级测试。
4. OpenAPI 兼容性检查。
5. 依赖、密钥、SAST 和容器扫描。
6. 构建前端和 API/Worker 镜像。

主分支到 staging：自动迁移、部署、健康检查、E2E、固定 RAG 冒烟。生产：人工批准、变更摘要、备份检查、渐进发布、监控观察和回滚判据。

## 12. 日志、指标与追踪

所有 HTTP 请求和任务使用 `request_id/trace_id`；Outbox 事件传递 trace。JSON 日志最少含：环境、服务、版本、模块、事件、耗时、结果、错误类别、tenant/company 的不可逆诊断标识。

关键指标：

- HTTP QPS、错误率、P50/P95/P99、限流次数。
- 数据库池、慢查询、锁、存储、RLS 拒绝。
- 队列深度、任务延迟、重试、死信和索引失败。
- 模型首 token/总延迟、错误、token、成本、拒答、Provider 切换。
- RAG 命中、证据门控、知识缺口、评测趋势。
- 业务漏斗：访问→有效对话→留资→跟进；只用定义好的事件口径。

告警必须指向运行手册并有 Owner，避免只有“红灯”没有处理动作。

## 13. 安全基线

- TLS、严格 CORS/CSP、管理端 CSRF 策略、Cookie 安全属性。
- 对象存储私有桶和短期签名 URL；禁止猜测式公开 key。
- 公共端点按 IP/卡片/访客会话多维限流；留资提供可插拔验证码/风控。
- 输入长度、文件白名单、病毒扫描、解析隔离。
- 密钥轮换、最小云权限、生产访问 MFA 和离职撤权。
- PII 加密和脱敏，敏感读取/导出/删除审计。
- Prompt injection、间接注入、跨租户检索和模型输出泄密测试。
- 定期依赖补丁、备份恢复和事故演练。
