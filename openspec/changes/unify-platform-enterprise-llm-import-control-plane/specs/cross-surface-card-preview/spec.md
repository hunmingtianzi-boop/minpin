## ADDED Requirements

### Requirement: 企业官方名片与员工名片是不同业务类型
系统 SHALL 为每张名片保存明确且不可含混的业务类型：`enterprise` 表示归企业所有的官方名片，`employee` 表示归具体员工所有的个人名片。企业官方名片 MUST 不依赖员工所有者；员工名片 MUST 绑定当前企业的有效成员。平台和企业端的允许列表投影 MUST 返回该类型。

#### Scenario: 企业管理员创建官方名片
- **WHEN** 企业管理员在企业名片区域创建名片
- **THEN** 系统创建不绑定员工的企业官方名片，并允许其独立编辑、预览、发布和下线

#### Scenario: 企业管理员创建员工名片
- **WHEN** 企业管理员在员工名片区域创建名片
- **THEN** 系统要求选择当前企业的有效员工，并创建与该员工绑定的员工名片

#### Scenario: 名片主人查看名片
- **WHEN** 名片主人进入本人工作区
- **THEN** 系统只返回其本人拥有的员工名片，不授予企业官方名片管理权限

### Requirement: 平台和企业端逐张展示名片
平台企业详情与企业名片管理 SHALL 按企业官方名片和员工名片分组展示企业范围内每一张名片的类型、名称/所有者、状态、更新时间和允许的管理动作，不得假设或推断“主名片”。

#### Scenario: 企业有多张名片
- **WHEN** 平台或企业用户查看该企业名片
- **THEN** 页面按稳定顺序展示全部名片及各自状态

### Requirement: 公开页链接只由服务端生成
只有状态为已发布且 API 返回 `share_url` 的名片 SHALL 显示“打开公开页”；前端不得根据 slug、租户或当前 host 自行拼接公开地址。

#### Scenario: 打开已发布名片
- **WHEN** 用户选择已发布名片的“打开公开页”
- **THEN** 浏览器以新标签和 `noopener noreferrer` 打开服务端返回的真实公开页

#### Scenario: 草稿或停用名片
- **WHEN** 名片未发布、已停用或没有 `share_url`
- **THEN** 页面不显示可点击公开链接，并明确说明当前没有公开页

### Requirement: 公开接口只返回公开内容
公开名片端 MUST 继续以当前 `/c/:slug` 页面和 `/api/v1/public/cards/{slug}` 数据合同为准，未发布内容与企业私密字段不得被公开访问。

#### Scenario: 访问未公开名片
- **WHEN** 匿名访客请求草稿、停用或不存在的名片 slug
- **THEN** 系统返回统一不可用结果且不泄漏后台状态或企业私密信息
